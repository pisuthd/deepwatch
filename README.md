## What is DeepWatch?

DeepWatch turns DeepBook's institutional-grade signals into something a human (or an agent) can act on in seconds.

[DeepBook](https://docs.sui.io/onchain-finance/deepbook-predict/) is Sui's central-limit-order-book exchange and prediction market. Its **Block-Scholes oracle** and **sub-400ms settlement** make it fast and accurate, but the raw signal is dense: SVI surfaces, multi-venue odds, time-decayed strikes. The information exists; the clarity doesn't.

DeepWatch solves that with three things no other DeepBook frontend ships together:

1. **A unified trading terminal** for Spot, Margin, and Predict in one UI — pick a product from the sidebar, trade, and never lose context.
2. **AI insights**, generated by Frontier AI (Claude 4.6 / MiniMax M3). The **SVI surface** (DeepBook Predict's own vol model — forward-vs-spot basis, vol regime, skew direction, SVI-vs-DB orderbook gap) is the **primary** directional signal; Polymarket and Kalshi are kept as a **secondary** sanity check that can adjust confidence and position size. Insights are written to **Walrus** and **encrypted with Seal** so the gate is the protocol, not a server.
3. **A stake-to-access pool** (Move smart contracts on Sui) that mints a non-transferable `Subscription` NFT and unlocks the encrypted slice — pro-rata yield for stakers, time-bounded access, on-chain expiry.

DeepBook Predict is powerful, but its strike-selection flexibility creates decision friction — traders hesitate, decisions slow down, and good entries get missed. DeepWatch removes that friction with AI-driven execution, Auto-Trade for high-confidence signals, and a PLP-based Subscription NFT that unlocks premium insights and yield opportunities.

## Architecture

### Two ways to trade on Predict

The /app/predict page offers two ways to mint a DeepBook Predict position. Both pull strikes and ranges from the same on-chain **SVI surface** — only who picks the numbers differs.

|  | Simple mode | Advanced mode |
| --- | --- | --- |
| **Binary** (single strike) | 5 default strikes, equally spaced ±5 ticks around spot, auto-bucketed from the SVI surface. | Pick any custom strike and bet UP or DOWN directly. |
| **Range** (price bracket) | 3 ranges auto-generated by the system around spot. | Drag the upper and lower lines to define the exact range to bet. |

### The data flow

**1. Snapshot.** Every 90 seconds the Compare page pulls live DeepBook oracles (Spot / Margin / Predict) and pairs each one with its closest Polymarket + Kalshi match by expiry. The snapshot is the AI batch.

**2. Generate.** The batch is streamed through an SSE consumer (`/api/insights/generate`) backed by MiniMax M3 (Anthropic-compatible). Each market gets a `MatchAnalysis`: signal (`UP` / `DOWN` / `NEUTRAL`), confidence, position size, reasoning, plus optional SVI and cross-venue citations.

   The decision framework is **SVI-primary, cross-venue-secondary**:

   - **A. SVI primary axis** — derive forward-spot basis, vol regime (low/normal/high/extreme), skew direction (`put_skew` / `call_skew` / `flat` from `b * rho`), and the SVI-vs-DB orderbook gap. The SVI direction is the call; cross-venue can only adjust confidence and size.
   - **B. NEUTRAL trigger** — `|basis| < 0.2%` AND `|SVI-vs-DB gap| < 2pp` → forced flat regardless of cross-venue.
   - **C. Cross-venue sanity check** — `dbVsConsensusPp` only downgrades confidence (`−0.1`) and halves position size when it strongly contradicts the SVI direction; it never flips direction.
   - **D. Position-size ladder** — SVI directional + cross-venue confirms → 6–8% bankroll; silent → 3–5%; contradicts → 1–2%; NEUTRAL → 0%.
   - **E. Confidence ladder** — base 0.5 if SVI present + direction, else 0.4; +0.15 if `|SVI-vs-DB gap| ≥ 10pp`; +0.10 cross-venue confirms; −0.10 cross-venue contradicts; −0.10 extreme vol regime; −0.20 SVI null. Clamp [0.0, 0.95].

   A **server-side consistency guard** (`sviDirectionConsistentWithSignal` in `lib/svi-signals.ts`) defends the core invariant — *put_skew + negative basis + DB-rich-on-UP ⇒ DOWN or NEUTRAL, never UP* — by demoting misfires to NEUTRAL with a visible `[SVI guard: reason]` suffix on the reasoning string and a `console.warn` for ops visibility.

**3. Persist with access control.** The full set is wrapped with Seal (AES-256-GCM bulk payload, Seal-wrapped AES key under the pool's namespace). The first 3 markets stay plaintext as a free "preview slice" — anyone can read them, including non-stakers, so the AI's directional read is public. The encrypted slice is only recoverable by a wallet holding an unexpired `Subscription` NFT for the matching pool.

### The on-chain access gate

```
User wallet (Sui)                Subscription NFT              Seal key server
        │                              │                              │
        │  holds Subscription          │                              │
        │  ─────────────────────► seal_approve(id, &sub, &clk)        │
        │                              │                              │
        │                              ├──► pool_id prefix check       │
        │                              ├──► clock < expires_at_ms      │
        │                              ├──► sender == sub.owner        │
        │                              │                              │
        │                              │◄─── key share (TEE-attested) ─┤
        │                              │                              │
        │  AES-decrypt full payload ◄──┤                              │
        │                              │                              │
```

The `seal_approve` entry function lives in [`contracts/sources/subscription.move`](contracts/sources/subscription.move) and is called by the Seal key server inside a simulated PTB — the key server refuses to release its share if the PTB aborts.

## Quick start

Prereqs: Node 20+, a Sui wallet (Slush, Sui Wallet, or any wallet-standard-compatible extension), and a Tatum API key (free tier works for Walrus upload/list; Polymarket + Kalshi search are public endpoints).

```bash
git clone https://github.com/pisuthd/deepwatch
cd deepwatch/frontend
npm install
cp .env.example .env       # then fill in your keys — see the table below
npm run dev
```

Open `http://localhost:3000`. The dev server uses HMR.

### Environment variables

| Var | Purpose |
| --- | --- |
| `NEXT_PUBLIC_TESTNET_GRPC` | Sui testnet gRPC endpoint |
| `NEXT_PUBLIC_MAINNET_GRPC` | Sui mainnet gRPC endpoint |
| `NEXT_PUBLIC_TATUM_API_KEY` | Tatum API key — used for Walrus upload/list |
| `MINIMAX_API_KEY` | MiniMax Anthropic-compatible API key (insight generation) |
| `MINIMAX_BASE_URL` | MiniMax base URL — override only if you proxy |
| `MINIMAX_MODEL` | MiniMax model id. Default: `MiniMax-M3` |
| `MINIMAX_THINKING_BUDGET` | Optional extended-thinking budget in tokens. Set to `0` to disable. Default: `2048` |

### Run the contracts

```bash
cd contracts
sui move build
sui move test
sui client publish --gas-budget 200000000
```

The first publish creates the `PoolCap` (one-shot, used by the demo admin functions). Subsequent `pool::init` calls create a fresh `Pool<PLP, SUI>` and bind it to the dapp's `NEXT_PUBLIC_DEEPWATCH_POOL_OBJECT_ID`.
